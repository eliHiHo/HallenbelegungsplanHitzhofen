## Project Overview

**Hallenbelegungsplan** is a web-based gymnasium hall booking management system for a German municipality. It replaces static PDF booking schedules with a dynamic online calendar where clubs can request training times and the municipality can approve bookings.

**Tech Stack:**
- **Backend:** Quarkus 3.32.4 (Jakarta REST, Hibernate Panache, PostgreSQL JDBC)
- **Frontend:** React SPA (empty/not yet started)
- **Database:** PostgreSQL 16
- **Deployment:** Docker + Caddy reverse proxy
- **Security Focus:** The project emphasizes web security (authentication, authorization, input validation, password hashing with bcrypt/jBCrypt)

## Build, Test, and Development Commands

### Backend (Quarkus/Maven)

All commands run from `/backend` directory:

**Development:**
```bash
./mvnw quarkus:dev
```
Starts dev mode with live coding at `http://localhost:8080`. Dev UI available at `http://localhost:8080/q/dev/`. Swagger UI at `http://localhost:8080/q/swagger-ui/`.

**Testing:**
```bash
./mvnw test                           # Run all unit tests
./mvnw test -Dtest=AuthServiceTest    # Run single test class
./mvnw test -Dtest=AuthServiceTest#login_succeeds  # Run specific test method
```

**Build:**
```bash
./mvnw package                        # Create JAR (quarkus-run.jar in target/quarkus-app/)
./mvnw package -Dquarkus.package.jar.type=uber-jar  # Create über-jar
./mvnw package -Dnative               # Create native executable (requires GraalVM)
./mvnw package -Dnative -Dquarkus.native.container-build=true  # Native build in container
```

**Database Migrations:**
Database schema is auto-generated by Hibernate ORM. Configuration in `src/main/resources/application.properties`:
- Dev: `quarkus.hibernate-orm.database.generation=drop-and-create` (fresh start each boot)
- Prod: `quarkus.hibernate-orm.database.generation=validate` (requires manual migration)

Initial test data loads from `src/main/resources/import.sql` (two pre-created users for testing).

**Configuration Files:**
- `pom.xml` - Maven dependencies and build config
- `src/main/resources/application.properties` - All config (DB, mail, CORS, logging, sessions)

### Docker

From root directory:
```bash
docker-compose -f docker/docker-compose.yml up
```
Starts PostgreSQL, backend, and Caddy reverse proxy. Requires `.env` with mail server credentials.

## High-Level Architecture

### Hexagonal Architecture Pattern

The backend strictly follows **hexagonal (ports & adapters) architecture** with clear separation of concerns:

```
┌─ Adapters (Interface Layer)
│  ├─ IN: REST API → DTOs → API Mappers → Use Cases
│  └─ OUT: Repository/Service Adapters → Domain Model
│
├─ Application Layer (Use Cases)
│  └─ Domain Services (stateless, transaction-scoped)
│
└─ Domain Layer (Core Business Logic)
   ├─ Models (immutable value objects)
   ├─ Ports (interfaces to external systems)
   └─ Exceptions (domain-specific errors)
```

### Key Domain Concepts

**Core Entities:**
- **User** (ADMIN, CLUB_REPRESENTATIVE): Authenticated users with roles and session management
- **Hall** (FULL, PART_SMALL, PART_LARGE): Configurable gymnasium sections
- **Booking**: Individual time slot bookings (has status: APPROVED, REJECTED, PENDING, CANCELLED)
- **BookingSeries**: Recurring bookings that generate individual Bookings
- **BookingRequest/BookingSeriesRequest**: Workflow for clubs to request slots and admin to approve
- **BlockedTime**: Admin-created time blocks (blackout periods for maintenance, events, etc.)

**Key Workflows:**
1. **Session Management:** Login → creates HTTPOnly cookie-based session (30-min timeout) → logout invalidates
2. **Booking Workflow:** Club requests → Admin approves/rejects → Creates individual Bookings
3. **Conflict Detection:** Blocks check overlaps with other bookings, blocked times, and public holidays
4. **Rate Limiting:** Failed login attempts trigger exponential backoff (in-memory store)

### Data Flow Example: Booking Creation

```
REST POST /bookings/request
  ↓
BookingRequestResource.createRequest(DTO)
  ↓
BookingRequestApiMapper.toDomain()
  ↓
BookingRequestService.createRequest()  [implements CreateBookingRequestUseCase]
  ↓
Domain validation: conflicts, permissions, hall availability
  ↓
BookingRepositoryPort.save(booking)  [outbound port]
  ↓
JpaBookingRepository.save()  [adapter implements port]
  ↓
BookingPersistenceMapper.toEntity()  [domain → DB model]
  ↓
EntityManager.merge()  [Hibernate persistence]
  ↓
NotificationPort.notify()  [outbound port, triggers email]
  ↓
BookingPersistenceMapper.toDomain()  [entity → domain]
  ↓
Response DTO
```

### Testing Patterns

Tests use **Mockito** for port mocking. Domain services are tested with in-memory implementations of repository ports. Example:

```java
class AuthServiceTest {
    static class InMemoryUserRepo implements UserRepositoryPort { ... }
    
    @Test
    void login_succeeds() {
        InMemoryUserRepo repo = new InMemoryUserRepo();
        AuthService service = new AuthService(repo, passwordVerification, ...);
        AuthSessionView result = service.login("user@example.com", "password");
        assertTrue(result.sessionId() != null);
    }
}
```

Run tests with: `./mvnw test -Dtest=AuthServiceTest#login_succeeds`

## Important Non-Obvious Patterns

1. **DTO Records:** All DTOs are immutable records (Java records), not classes. This enforces immutability and reduces boilerplate.

2. **Mapper Layering:** Three mapper types exist:
   - `*ApiMapper` - REST DTO ↔ Domain Model
   - `*PersistenceMapper` - Domain Model ↔ JPA Entity
   - Mappers have constructors to inject dependencies (other mappers for nested objects)

3. **Port Dependencies:** Domain services depend only on outbound port interfaces, never on concrete implementations. Concrete adapters are instantiated in repositories using constructor injection.

4. **Email/Notification:** Password reset emails use `PasswordResetMailPort` (adapter: `QuarkusPasswordResetMailAdapter`). Template URLs reference `app.base-url` config for frontend password reset page.

5. **Public Holiday Integration:** `BavariaPublicHolidayAdapter` checks for Bavarian holidays when detecting booking conflicts.

6. **Conflict Validation:** Bookings check against:
   - Other overlapping Bookings (same hall, time overlap)
   - BlockedTime entries (admin blackout periods)
   - Public holidays (Bavaria calendar)
   - Hall availability config (opening hours, max duration, etc.)

7. **Role-Based Access:** Services check `user.isAdmin()` or `user.isClubRepresentative()` and throw `ForbiddenException` for unauthorized operations.

8. **Error Handling:** `ApiExceptionMapper` catches domain exceptions and converts them to HTTP responses:
   - `ValidationException` → 400 Bad Request
   - `NotFoundException` → 404 Not Found
   - `UnauthorizedException` → 401 Unauthorized
   - `ForbiddenException` → 403 Forbidden
   - `RateLimitException` → 429 Too Many Requests
   - `BookingConflictException` → 409 Conflict

9. **Clock Injection:** Uses `java.time.Clock` (injected by Quarkus) instead of `Instant.now()`, enabling time-based testing.

10. **Session Validation:** Every request needs a valid session cookie. Sessions are tied to users and invalidated on logout or when a new login occurs for the same user.
